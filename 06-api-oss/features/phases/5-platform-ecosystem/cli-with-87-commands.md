# CLI with 87 Commands

## What It Does
A full-featured command-line interface exposing 87 total commands built with Rust's clap
derive macro. 38 top-level commands and 49 nested subcommands across 9 subcommand enums:
GraphCmd (5 subcommands: query, add, delete, traverse, stats), LedgerCmd (5: verify, export,
import, status, prune), AuthCmd (7: login, logout, status, create-token, revoke, list-
sessions, rotate-keys), ServiceCmd (5: install, uninstall, start, stop, status), SyncCmd
(5: start, stop, status, add-peer, remove-peer), BackupCmd (2: create, restore), FinetuneCmd
(3: start, status, list), CacheCmd (2: clear, status), and ModelCmd (15: pull, list, info,
remove, quantize, benchmark, serve, stop-serve, download, verify, set-default, show-config,
search, prune-local, export, import). Operates fully offline with zero cloud dependencies.

## How It Works
The CLI is implemented in `main.rs` under the workspace root. It uses `clap` derive macros
to define the argument parser with nested enums — each of the 9 subcommand groups is a
separate enum with clap derive attributes for auto-generated help, shell completions, and
man pages. Each command maps to a WebSocket message in the 810-message protocol — the CLI
sends a `ClientMessage` variant over `tokio-tungstenite` to the gateway on port 3030 and
receives a `ServerMessage` response with the result. The CLI supports both local mode
(connecting to `ws://localhost:3030`) and remote mode (via the public URL tunnel with TLS).
The `--help` output is auto-generated by clap with examples and argument descriptions.
Command output is formatted as JSON by default (for piping and composing with `jq`, `yq`,
`jmespath`) with a `--pretty` flag for human-readable colored output. The CLI reads
configuration from `opencode.json` at the current directory or parent directories, merging
with gateway-level config. All commands include a `--timeout` flag and respect the
`API_OSS_HOST` and `API_OSS_PORT` environment variables. The CLI binary is the same ~81 MB
statically-linked binary that includes the engine, WebSocket server, REST API, bridge
agents, and tunnel — no separate CLI installation needed. The 87 total commands provide
comprehensive coverage of: graph operations, ledger management, authentication and
authorization, model lifecycle (pull, quantize, benchmark, prune), service management
(systemd/launchd/Windows service), P2P sync, backup and restore, fine-tuning, cache
management, bridge control (Telegram/Discord/WhatsApp), migration, compliance, total cost
of ownership reporting, MCP server control, and system diagnostics. Common flag patterns
include `--format` (json, yaml, table), `--output` (file path), `--timeout` (seconds,
default 30), and `--pretty` (colored human-readable output). For example,
`api-oss start --browser --port 8080` launches the gateway and opens the web UI,
`api-oss graph query --type Document --limit 50 --format json | jq '.nodes | length'`
pipes graph results through jq, and `api-oss model pull qwen2-vl-2b-q4 --progress` downloads
a model with a live progress bar. Shell completion scripts are auto-generated via
`api-oss completion bash` (or zsh, fish, powershell) and can be sourced in the user's shell
profile for tab-completion of all 87 commands and their flags.

## How to Operate
1. Ensure the gateway is running: `api-oss start` or run the binary directly. The CLI
   connects to the gateway on port 3030 automatically.
2. Get top-level help: `api-oss --help` — shows all 38 top-level commands organized by
   category with brief descriptions.
3. Get subcommand help: `api-oss graph --help` shows the 5 graph subcommands with options.
4. Graph operations: `api-oss graph query --type person --limit 100`,
   `api-oss graph add --label "Person" --property name=John`.
5. Ledger operations: `api-oss ledger verify`, `api-oss ledger export --format json`.
6. Model management: `api-oss model pull qwen2-vl-2b-q4`, `api-oss model list`,
   `api-oss model benchmark`.
7. Finetuning: `api-oss finetune start --dataset ./data.jsonl --base-model qwen2-vl-2b-q4`.
8. Sync: `api-oss sync add-peer --address 192.168.1.100:3030`, `api-oss sync status`.
9. Backup: `api-oss backup create`, `api-oss backup list`.
10. Service: `api-oss service install`, `api-oss service status`.
11. Bridge: `api-oss bridge telegram status`, `api-oss bridge discord connect`.
12. Other: `api-oss doctor` (system diagnostics), `api-oss compliance` (compliance check),
    `api-oss tco` (total cost of ownership report).
13. Use `api-oss start --browser --port 8080` to start the gateway and automatically open the
    web UI in the default browser on a custom port. Combine flags:
    `api-oss start --browser --tunnel --model-path ./models/qwen.gguf`.
14. Pipe commands into each other: `api-oss graph query --type Person --format json | jq '.nodes
    | length'` to count nodes. Use `--format yaml` for human-readable output in CI logs.
15. Enable shell completions: eval `$(api-oss completion zsh)` (zsh) or add
    `Invoke-Expression (& { (api-oss completion powershell | Out-String) })` to PowerShell
    $PROFILE for tab-completion of all 87 commands, flags, and subcommand arguments.

## The Moat
- Building 87 commands with comprehensive argument parsing, nested subcommand enums, and
  full offline capability represents years of engineering that cannot be replicated quickly.
- Every command connects to the local gateway via WebSocket — no REST endpoints to version,
  no API keys to manage, no cloud services to authenticate with.
- The clap derive macros generate `--help` output, shell completions (bash, zsh, fish,
  PowerShell), and man pages automatically — zero documentation maintenance overhead.
- All commands work offline with no internet required — unique among AI platform CLIs where
  every operation requires cloud API calls.

## Why Choose API-OSS
A DevOps engineer can script an entire deployment pipeline using `api-oss service install &&
api-oss backup schedule && api-oss sync add-peer` — all without a browser or cloud
dependency. A data scientist can extract graph data with `api-oss graph query --format json
| jq` and pipe it into their analysis tools. A system administrator can diagnose issues
with `api-oss doctor` and check compliance with `api-oss compliance`. No other AI decision
platform offers a CLI of this depth running fully offline from a single ~81 MB binary.

## Competitive Comparison
- **Palantir**: Limited CLI, cloud-dependent, few commands. Most operations require Foundry
  web UI access.
- **OpenAI**: Stateless CLI (`openai`), requires API key, only ~10 commands. No graph,
  ledger, sync, or backup commands.
- **Anthropic**: Minimal CLI (`claude`), cloud-only, few commands. No offline capability.
- **Google**: gcloud CLI is for cloud infrastructure, not an AI decision engine. Hundreds
  of commands but none for knowledge graphs or sovereign AI operations.
- **Nvidia**: nvidia-smi is hardware monitoring only. nvidia-cli is inference-focused, not
  a decision engine CLI.
- **Snowflake**: snowsql is SQL-only, no graph/ledger/agent/model/backup commands.

## Cost-Benefit Analysis
OpenAI CLI requires API calls costing $0.15/M tokens — a single `graph query` returning
10k nodes would cost nothing locally but could cost hundreds via cloud API. Palantir CLI
requires a Foundry license ($500k+/year). Snowsql queries cost Snowflake credits
($2–$8/credit). API-OSS CLI costs $0 — every one of the 87 commands runs locally with no
usage fees. The time saved by scripting automation with `api-oss` CLI (vs manually clicking
through a UI) is ~1 hour/day for power users, saving $25k/year at $100/hour. Shell
completion and auto-generated help reduce learning time from days to minutes.
ngrok charges $20/month for tunnel access to expose local services; API-OSS includes a
built-in free tunnel with Let's Encrypt TLS, saving $240/year per developer. OpenAI CLI
usage costs $0.01/1K tokens for inference — a data scientist running 50 CLI queries/day
at 2K tokens each pays $1/day or $30/month. API-OSS CLI runs every command locally at $0
inference cost, eliminating this recurring expense entirely.
