# Regression Testing

## What It Does
Captures baseline model responses for a curated set of queries.
Compares new responses against baselines to detect regressions.
Automated quality gates block model promotion if regressions exceed thresholds.
Baselines are stored in the ledger with full version history for traceability.

## How It Works
The gateway starts via `api-oss start`, loading regression testing configuration
from `opencode.json` under the `regression_testing` section.
The `regression_test.rs` Rust module in `ai-oss-gateway/src/` manages baselines
and comparisons.
On first setup, an operator defines a set of canonical test queries — typed
manually, imported from a JSONL file, or generated by recording actual user
queries from production traffic.

Each query is run against the current production model, and the response is
captured as the baseline. The module deduplicates queries by computing a
SHA-256 of the normalized query string (lowercased, whitespace-trimmed), so
re-adding an identical query updates the baseline rather than creating a
duplicate.
Baselines are stored in the ledger at `./data/ledger/regression_baselines/`
with the model hash, query hash, response, timestamp, and a SHA-256 of the
full baseline entry. Each baseline file is a JSON object with fields:
`model_hash`, `query`, `query_hash`, `response`, `metrics` (initial scores),
and `timestamp`.
When a new model version is proposed for promotion, the gateway runs the same
queries against the candidate model. The run is parallelized using a
threadpool — 4 concurrent inferences by default, configurable via
`regression_testing.concurrency` in `opencode.json`.

For each query, the response is compared to the baseline using configurable
metrics: exact match, BLEU score, ROUGE-L, semantic similarity via cosine
distance on embeddings (computed using the same qwen2-vl-2b-q4 model's
embedding layer), and custom evaluator functions registered as WASM plugins.
Each metric is assigned a weight in the composite score (defaults: BLEU 0.3,
ROUGE-L 0.3, semantic similarity 0.4), and a per-query pass/fail threshold
(default 0.8 composite).
Per-query comparison results are aggregated: the module computes the number
of passes, failures, and regressions (scores decreased beyond a threshold).
If the regression rate exceeds the configured threshold — e.g., "no more than
5% of queries may regress" — the promotion is blocked and the quality gate
pipeline emits a `regression_test_failed` event on the WebSocket.

Operators can inspect every regression at the query level, viewing side-by-side
baselines vs. new responses with highlighted differences. The diff view uses a
token-level alignment algorithm to highlight insertions, deletions, and
substitutions between the baseline and candidate response.
The frontend `RegressionTestView` connects via WebSocket to port 3030 and
provides an interactive regression dashboard with real-time updates as each
query comparison completes.
The view shows pass/fail rates, trend charts over model versions (line chart
of composite score per model hash), and a query-level drill-down that displays
the per-metric breakdown for each query.

Regression test runs are triggered manually via WS message
`regression_test_run` or automatically as part of the quality gate pipeline.
Results are returned via `regression_test_result`.
Scheduled regression runs can be configured with a cron expression in
`opencode.json`.
The HTTP UI on port 8081 displays a regression summary with color-coded status
indicators.

All regression test results are committed to the ledger for auditability,
including the candidate model hash, baseline model hash, per-query scores, and
the overall pass/fail decision.
The 87 CLI commands include `model regression-run`, `model
regression-baseline-set`, and `model regression-status`.

## How to Operate
1. Set initial baselines: `api-oss model regression-baseline-set
   --queries ./regression_queries.jsonl --model production`.
2. Add a single query: `api-oss model regression-baseline-add --query "..."
   --id q1 --model production`.
3. List baselines: `api-oss model regression-baseline-list --model production`.
4. Run regression tests: `api-oss model regression-run --candidate my_finetune
   --baseline production`.
5. View results: `api-oss model regression-status --run <run_id>`.
6. Export results: `api-oss model regression-export --run <run_id> --format
   json --output ./data/regression_report.json`.
7. In the frontend on port 8081, navigate to Model Lifecycle > Regression
   Tests to see pass rate, trend chart, and query-level drill-down.
8. Configure automatic regression gates in `opencode.json`:
   ```json
   {
     "quality_gates": {
       "gates": [
         { "metric": "regression_pass_rate", "op": ">=", "threshold": 0.95 }
       ]
     }
   }
   ```
6. Add a query: `api-oss model regression-baseline-add --query "..." --id q1`.
7. Schedule: configure `"regression_schedule_cron": "0 */6 * * *"`.

## The Moat
- All regression testing runs locally — no API calls to external services
- Baselines stored in the ledger with full version history and rollback
- Multiple comparison metrics provide comprehensive regression detection
- Per-query granularity enables targeted investigation
- Integration with quality gates blocks promotion on regressions
- No cloud test infrastructure required

## Why Choose API-OSS
Customers choose API-OSS regression testing for production-grade quality
assurance without cloud dependency.
Palantir's regression testing requires Foundry's cloud orchestration.
OpenAI offers no regression test tooling.
API-OSS bakes regression testing into the model lifecycle, making it impossible
to accidentally degrade model quality.

## Competitive Comparison
- **Palantir**: Regression testing exists but requires Foundry's cloud
  orchestration.
- **OpenAI**: No regression test tooling for fine-tuned models.
- **Anthropic**: No regression testing tooling.
- **Nvidia**: No regression testing for local models in NeMo.
- **Mercor**: No model regression testing.

## Cost-Benefit Analysis
Cloud CI/CD regression testing costs ~$500/month in runner time and API costs
for 100-query suites — equivalent to $6K/year.
API-OSS runs regression tests locally at $0 additional infrastructure cost.
A single undetected regression costs an average of ~$15K in incident response,
debugging, rollback, and customer impact.
Catching 1 regression incident per quarter saves $60K/year.
Manual comparison of 50 queries takes 2 hours per run at $100/hour operator
cost — API-OSS automates to seconds, saving $200 per run.
With weekly regression runs, this saves $10,400/year in operator time alone.

## Applications
- **Consumer**: Catch regressions when updating personal models.
- **Government / Defense**: Guarantee critical capabilities are never degraded.
- **Enterprise**: Automated quality gates in CI/CD pipelines for model
  deployment.
