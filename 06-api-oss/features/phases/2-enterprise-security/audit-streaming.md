# Audit Streaming

## What It Does
Real-time audit event stream that captures every auth decision, graph mutation,
configuration change, and rule evaluation.
Integrates with external SIEM systems via syslog for continuous compliance monitoring and
provides a structured, immutable audit log with cryptographic chaining.
## How It Works
The audit streaming pipeline is embedded directly in the Rust engine at
`ai-oss-gateway/src/`.
Every auditable action — auth success or failure, role changes, graph node
create/update/delete, configuration changes, rule violations, WebSocket connections, and
PSI sessions — generates a structured JSON event before the response is sent to the
client.
The Rust event dispatcher writes each event to two destinations in parallel: the immutable
ledger at `data/ledger/` (`.aioss` format with SHA-256 chaining for tamper detection) and
an in-memory ring buffer for real-time streaming.
Frontend views subscribe to the real-time stream via WebSocket to port 3030 by sending a
`stream_audit` message.
The HTTP UI served on port 8081 displays the `AuditStreamView` for live monitoring.
For SIEM integration, events are forwarded via syslog (RFC 5424) to external systems like
Splunk, QRadar, Elastic, or any syslog-compatible receiver.
Syslog forwarding is configured in `opencode.json` with optional TLS and bearer token
authentication (tokens generated via `api-oss auth generate`).
The gateway is started via `api-oss start` or the binary directly.
Every event includes: timestamp (RFC 3339 nanosecond precision), user identifier, action
type, resource identifier, outcome (allow/deny/error), request ID for correlation, and a
SHA-256 hash of the previous ledger entry for chaining.
The CLI (`api-oss auth log --tail`) provides terminal-based access to the audit stream,
one of 87 commands across 9 subcommand groups (auth, service, sync, backup, etc.).
All of this operates fully air-gapped with no internet — critical for classified and
regulated environments.
## How to Operate
1.
**Open the AuditStreamView** at `https://localhost:8081/audit` to see live events as they are generated by the Rust engine.
2.
**Subscribe programmatically** by sending a WebSocket message `{"type": "stream_audit"}` to `wss://localhost:3030`.
Events stream in real time as JSON objects.
3.
**Configure syslog forwarding** in `opencode.json` under `audit.syslog` with target host, port, protocol (TCP/UDP), and optional TLS settings.
4.
**Filter events** in the frontend by action type, user, resource, or outcome.
The UI supports saved filters for common queries.
5.
**Search the immutable ledger**: Use `api-oss audit query --from 2025-01-01 --to 2025-06-01 --user alice` to query historical events.
The CLI accesses `data/ledger/*.aioss` files directly.
6.
**Verify ledger integrity**: Run `api-oss audit verify` to check the SHA-256 chain from the genesis entry to the latest event.
Any break in the chain indicates tampering.
7.
**Export for compliance**: The frontend supports exporting event ranges as CSV, JSON, or PDF for auditor review.
## The Moat
- **No audit gaps**: Events are emitted from the Rust engine before the response is sent.
There is no polling, no batch export, no log file parsing.
Every auditable action generates a guaranteed event.
- **Cryptographically chained ledger**: The `data/ledger/` directory stores events in `.aioss` format with each entry containing the SHA-256 hash of the previous entry.
Tampering with any event breaks the chain — detectable with a single CLI command.
- **Zero-latency streaming**: The in-memory ring buffer provides sub-millisecond event delivery to all subscribed WebSocket clients.
No database query needed to view the current event stream.
- **Air-gapped SIEM integration**: Syslog forwarding works without internet.
Events leave the air-gapped network only through the configured syslog channel —
typically a one-way data diode or secured log aggregator.
- **Pure Rust pipeline**: No Logstash, no Fluentd, no separate log shipping agent.
The audit pipeline is compiled into the same single binary started by `api-oss start`.
- **Immutable storage**: The ledger at `data/ledger/` is append-only.
Even the root user cannot modify or delete past entries without breaking the cryptographic
chain.
## Why Choose API-OSS
Cloud AI vendors provide audit logs through dashboards with limited retention and no
cryptographic integrity guarantees.
OpenAI's audit logs are delayed, limited to 90 days, and cannot be verified by the
customer.
Palantir's Foundry audit requires cloud connectivity.
API-OSS provides real-time, cryptographically chained audit streaming as a built-in
feature of an air-gapped single binary.
Everything works offline, events are immutable, and SIEM integration is built in.
For SOC 2, ISO 27001, or government compliance, API-OSS provides the audit infrastructure
that cloud vendors cannot match.
## Competitive Comparison
- **OpenAI**: Audit logs via API dashboard — delayed by minutes to hours, limited to 90-day retention, no cryptographic integrity.
No syslog export capability.
- **Palantir**: Audit via Foundry, cloud-dependent.
Events can be lost if network connectivity is interrupted.
No offline audit trail.
- **Snowflake**: Audit via Snowflake account usage views — query-based, not real-time streaming.
Limited to cloud-deployed instances only.
## Cost-Benefit Analysis
Building an equivalent audit pipeline requires: a log aggregation service like Splunk
($150-$1,000+/month per GB ingested), a separate immutable storage layer for cryptographic
chaining, and engineering time to build the event collection pipeline (2-4 months of
development).
Cloud vendors do not expose raw audit streams — customers are limited to what the
dashboard shows.
API-OSS provides a complete, cryptographically verified audit pipeline at zero additional
cost.
Time savings: 2-4 months of infrastructure development eliminated.
Risk reduction: immutable ledger guarantees audit integrity for regulatory compliance —
tampering is immediately detectable.
