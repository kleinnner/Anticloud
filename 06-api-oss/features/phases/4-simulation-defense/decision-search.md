# Decision Search

## What It Does
Full-text search over all decisions made by the system. Filter by
council,
agent, date range, outcome type, and semantic similarity to find
relevant
historical decisions. Search results include full context and linked
entities
from the knowledge graph.

## How It Works
The decision search module in
`ai-oss-gateway/src/decision_search.rs`
maintains dual indexes over the decision provenance graph. A
full-text index
(using Tantivy in Rust) indexes decision titles, descriptions, tool
call
inputs/outputs, and reasoning text. A semantic index stores
embedding
vectors generated by the Qwen2-VL-2B-Instruct-Q4_K_M.gguf model on
CUDA
for similarity search. When a query arrives via WebSocket on port
3030
(`decision_search` message), the search engine executes both a
lexical
search (BM25) and a semantic search (cosine similarity), then fuses
results
using reciprocal rank fusion. Results are ranked by relevance with
highlighted snippets. Filters narrow results by council name, agent
ID, date
range, outcome type (approved, rejected, escalated), confidence
range, and
related entities. The DecisionSearchView on the HTTP UI at port 8081
provides
an interactive faceted search interface with real-time filtering.
Results
link directly to the DecisionTreeView for full provenance
exploration. The
index is incrementally updated as new decisions are made, stored in
`./data/`. Config is driven by `opencode.json` under the
`decision_search`
section. The CLI includes 87 commands with `decision-search`
subcommands.

## How to Operate
1. Start the gateway: `api-oss start` or binary directly
2. Open the DecisionSearchView in the browser at port 8081
3. Enter a search query — natural language or keyword-based
4. Apply filters: council, agent, date range, outcome type, confidence range
5. Browse results with relevance scores and highlighted snippets
6. Click any result to open the full decision provenance tree
7. Use semantic search for concept-based queries — "find decisions about
   logistics failures"
8. Save frequent searches as named queries
9. Export search results as CSV or graph snapshots to `./data/`

## The Moat
- Searching over a provenance-rich decision graph with both lexical and
  semantic retrieval requires maintaining dual indexes over a
constantly
  growing decision history
- The integration with the graph means search results include full context
  and linked entities — not just text matches
- Semantic search via local LLM embeddings provides concept-level retrieval
  without cloud dependency
- Each result links directly to the full provenance tree, enabling drill-down
  from search to complete decision audit
- Incremental indexing ensures new decisions are immediately searchable
- No competitor offers search over a decision provenance graph at this scale

## Why Choose API-OSS
Palantir's search exists but is limited to structured data queries
and lacks
decision-specific indexing. Google has no equivalent decision search
product.
Anthropic has no decision search product. API-OSS provides
dual-index search
(lexical + semantic) over the complete decision provenance graph,
enabling
analysts to find relevant historical decisions by content, context,
or
concept — all offline on local hardware.

## Cost-Benefit Analysis
Implementing dual-index search (lexical + semantic) with graph
context
typically costs $200K–$500K in development plus ongoing cloud
embedding API
fees ($100–$1,000/month). API-OSS provides this capability at zero
software
cost — the Qwen2-VL-2B-Instruct-Q4_K_M.gguf model runs embedding
generation locally on CUDA with no API fees. A defense intelligence
organization searching across 1M+ historical decisions saves
$200K–$500K in
custom development. Time savings: manual search across decision logs
takes
hours; API-OSS returns results with provenance context in
milliseconds.
