# Kasteran* — CPU-GPU Heterogeneous Computing
© Lois-Kleinner & 0-1.gg 2026

## Overview

Modern systems contain both CPUs (optimized for sequential and latency-sensitive tasks) and GPUs (optimized for parallel throughput). Kasteran* provides automatic dispatch between CPU cores and GPU, allowing developers to leverage both without manual orchestration.

## Heterogeneous Architecture

Kasteran* treats the CPU and GPU as a unified compute resource:

```
                    +------------------+
                    |  Kasteran* Code  |
                    +--------+---------+
                             |
                    +--------+---------+
                    |  Compiler/Driver  |
                    +--------+---------+
                             |
              +--------------+--------------+
              |              |              |
     +--------+------+ +----+---------+ +--+--------+
     | CPU Cores    | | GPU CUs/SMs  | | Accelerator|
     | (latency opt)| | (throughput) | | (special)  |
     +--------------+ +--------------+ +------------+
```

## Automatic Dispatch

The Kasteran* compiler and runtime automatically determine where to execute code.

### Dispatch Criteria
The runtime considers:

- **Data parallelism**: Highly parallel work goes to GPU
- **Data size**: Small datasets stay on CPU (transfer overhead)
- **Latency requirements**: Low-latency work stays on CPU
- **Dependency depth**: Sequential chains stay on CPU
- **Memory access**: Random access stays on CPU; streaming goes to GPU
- **Previous execution**: Historical performance guides future decisions

### Compiler Analysis
The compiler performs static analysis to estimate suitability:

```
fn process_large_dataset(data: [f64]) -> [f64] 
    @dispatch(analysis = "data_parallel")
    @estimated_gpu_speedup(12.5x)
```

The annotation is generated by the compiler based on:
- Loop independence analysis
- Memory access pattern analysis
- Computational intensity estimation
- Data transfer cost estimation

## Memory Management

Heterogeneous computing requires careful memory management.

### Unified Memory
Kasteran* provides a unified memory model:

```
var data = allocate_shared_memory::<[f64]>(1_000_000)
// Accessible from both CPU and GPU
// Data migrates automatically as needed
```

### Explicit Control
For performance-critical paths, developers can control memory placement:

```
@gpu_memory
var gpu_data = [1.0, 2.0, 3.0]

@cpu_memory
var cpu_data = [1.0, 2.0, 3.0]
```

### Zero-Copy Access
Where possible, data is accessed without copying:
- Pinned memory for direct GPU access
- Mapped memory for CPU-GPU sharing
- Peer-to-peer access between GPUs

## Execution Model

### Kernel Launch
Parallel operations are compiled into GPU kernels:

```
let result = data.map(|x| x * 2.0)  // Auto GPU kernel
```

### Pipeline Execution
CPU and GPU execute in parallel:
```
// CPU prepares next batch while GPU processes current
let batch1 = load_data(0, 1000)
let result1 = process_gpu(batch1)    // GPU executes
let batch2 = load_data(1000, 2000)   // CPU prepares
```

### Synchronization
Synchronization is automatic at pipeline boundaries:
- Results are available when accessed
- Dependencies are tracked
- Barriers are inserted only when needed

## Performance Model

Kasteran* provides performance predictions:

| Operation | CPU Time | GPU Time | Speedup |
|---|---|---|---|
| Element-wise 1M | 2ms | 0.1ms | 20x |
| Matrix mult 1024 | 50ms | 2ms | 25x |
| Sort 10M | 200ms | 50ms | 4x |
| Sequential | 1ms | 10ms | 0.1x (CPU better) |

## Heterogeneous Scheduling

The runtime scheduler manages work distribution:

### Work Partitioning
Parallel work is partitioned between CPU and GPU:
- Large, regular partitions go to GPU
- Small, irregular remain on CPU
- Dynamic load balancing adjusts partitions

### Priority Scheduling
Real-time and latency-sensitive work gets CPU priority:
- Interactive response on CPU
- Background batch processing on GPU
- Mixed workloads are balanced

## Profiling and Optimization

Kasteran* provides profiling tools:

```
kasteran profile --heterogeneous my_app
```

This reports:
- CPU vs GPU time distribution
- Data transfer overhead
- Kernel launch latency
- Utilization percentages
- Speedup estimates

## Conclusion

Kasteran* heterogeneous computing capabilities automatically dispatch work between CPU cores and GPU, maximizing the utilization of existing hardware without requiring developers to manage the complexity of heterogeneous programming manually.
