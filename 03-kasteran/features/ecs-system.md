# Kasteran* ‚ÄĒ Entity-Component-System Architecture

¬© Lois-Kleinner & 0-1.gg 2026

## Overview

Kasteran* has a first-class Entity-Component-System (ECS) architecture built directly into the language. Unlike traditional ECS libraries that are bolted on via third-party crates, Kasteran* ECS is part of the compiler's semantic analysis phase, enabling compile-time component validation, cache-friendly layout generation, and automatic parallel system scheduling.

## Core Primitives

### `@+` ‚ÄĒ Spawn

Creates a new entity with attached components. The spawn rune accepts a name followed by a parenthesized list of component values.

```
@+ Player(
    Position(x: 0.0, y: 0.0),
    Velocity(dx: 5.0, dy: 0.0),
    Sprite(path: "assets/player.png")
)
```

Spawn returns an `EntityId` which can be stored for later use:

```
player_id := @+ Player(Position(x: 0.0, y: 0.0))
```

### `@>` ‚ÄĒ Query

The query rune retrieves entities matching a component signature. Queries are compile-time checked: the compiler verifies that every queried component type exists.

```
@> Position, Velocity {
    |+ update(pos: Position, vel: Velocity) {
        pos.x = pos.x + vel.dx * dt
        pos.y = pos.y + vel.dy * dt
    }
}
```

Query expressions support component filtering, iteration, and projection.

### `@~` ‚ÄĒ System

Declares a system that runs on a schedule. Systems are automatically parallelized when they operate on disjoint component sets.

```
@~ MovementSystem : Position, Velocity {
    |+ run(pos: Position, vel: Velocity, dt: f32) {
        pos.x = pos.x + vel.dx * dt
        pos.y = pos.y + vel.dy * dt
    }
}
```

## System Scheduling

Kasteran*'s compiler analyzes system component access patterns and generates an optimal execution schedule. Systems that write to disjoint component tables run in parallel.

```
@~ RenderSystem  : Position, Sprite   // reads Position, writes nothing
@~ PhysicsSystem : Position, Velocity // writes Position
```

The compiler detects that `RenderSystem` only reads `Position` while `PhysicsSystem` writes it, and schedules them sequentially. If they were both read-only, they would execute in parallel.

## Component Storage

Components are stored in contiguous arrays (archetype chunks). The compiler generates optimal struct-of-arrays (SoA) layout:

```
// Generated by the compiler:
struct ArchetypeChunk_PositionVelocity {
    positions: [Position; 64],
    velocities: [Velocity; 64],
    entities: [EntityId; 64],
}
```

This layout maximizes CPU cache utilization during system iteration.

## Example: Complete ECS Game Loop

```
use "std/io"

$$ Position = (x: f32, y: f32)
$$ Velocity = (dx: f32, dy: f32)
$$ Sprite   = (path: string)

@+ Player(Position(0.0, 0.0), Velocity(5.0, 0.0), Sprite("player.png"))
@+ Enemy(Position(100.0, 100.0), Velocity(-3.0, 0.0), Sprite("enemy.png"))

@~ Physics : Position, Velocity {
    |+ update(pos: Position, vel: Velocity, dt: f32) {
        pos.x = pos.x + vel.dx * dt
        pos.y = pos.y + vel.dy * dt
    }
}

@~ Renderer : Position, Sprite {
    |+ draw(pos: Position, spr: Sprite) {
        print("draw " + spr.path + " at " + pos.x.to_string())
    }
}

|+ main() -> i32 {
    dt :~ 0.016
    loop {
        @~ Physics { |system| system.run(dt) }
        @~ Renderer { |system| system.draw() }
    }
    => 0
}
```
