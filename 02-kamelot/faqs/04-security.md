## Is My Data Encrypted?

Yes. Kamelot encrypts all file content stored in the flat store using **XChaCha20-Poly1305 AEAD** (Authenticated Encryption with Associated Data).

### Encryption Details

| Parameter | Value |
|-----------|-------|
| Algorithm | XChaCha20-Poly1305 |
| Key size | 256 bits |
| Nonce size | 192 bits (XChaCha20 extended nonce) |
| Authentication tag | 128 bits (Poly1305) |
| Key derivation | HKDF-SHA256 (master key + inode number) |
| Encryption scope | Per-file (each file gets a unique derived key) |

### What Is Encrypted

| Component | Encrypted? | Algorithm |
|-----------|-----------|-----------|
| File content (flat store) | Yes | XChaCha20-Poly1305 |
| File metadata (ledger) | No (but integrity-protected) | SHA256 hashes |
| Vector embeddings (Qdrant) | No (vectors are not content) | — |
| Encryption keys | Yes | Master key encrypted with Argon2id-derived key |
| Configuration | Optional | — |

### Why Per-File Encryption?

Each file in the flat store is encrypted with a key derived from the master key and the file's inode number:

```
file_key = HKDF-SHA256(master_key, inode_number, "kamelot-file-encryption-v1")
```

This means:
- Compromising one file's key does not compromise other files
- Deleting a file's key (via master key rotation) renders it permanently unreadable
- Each file can be independently encrypted/decrypted without touching other files

### Encryption at Rest vs In Transit

| State | Protection |
|-------|-----------|
| At rest (flat store) | XChaCha20-Poly1305 encryption |
| At rest (original files) | Unchanged (your original files remain as they were) |
| In transit (local IPC) | Unix domain sockets (Linux/macOS) or named pipes (Windows) |
| In transit (network) | Not applicable (Kamelot does not network your data) |

## Who Has Access to My Encryption Keys?

**Only you.** Encryption keys are generated locally on your machine and never leave it.

### Key Hierarchy

```
User Password (optional layer)
    |
    v
Master Encryption Key (256-bit, generated by OS random)
    |
    ├── File Key 1 (derived via HKDF) → File 1
    ├── File Key 2 (derived via HKDF) → File 2
    ├── File Key 3 (derived via HKDF) → File 3
    └── ... (unique key per file)
```

### Key Storage

| Key | Storage Location | Protection |
|-----|-----------------|------------|
| Master key | `~/.kamelot/keys/master.key` | Encrypted with Argon2id (if password set) or OS keychain |
| File keys | Derived on-the-fly | Never stored on disk |
| Recovery key | User-provided backup | Printed during `kml init` — must be stored offline |

### What Kamelot Inc. Knows

**Nothing.** Kamelot Inc. has:
- No access to your encryption keys
- No access to your files
- No access to your file metadata (filenames, paths, sizes)
- No access to your vector embeddings
- No access to your query history

Kamelot is self-hosted, open source, and designed so that even if compelled, the company cannot access user data.

## Can Kamelot See My Files?

Kamelot processes your files locally to generate embeddings and content hashes. This processing is done entirely on your machine by the Kamelot daemon process.

### What Kamelot Processes

1. **File content**: Read from disk to parse text, extract metadata, and generate embeddings
2. **File metadata**: Filename, path, size, modification date, MIME type
3. **File structure**: For images, visual features; for documents, text structure

### What Happens to Processed Data

| Data Type | Stored Locally? | Transmitted? | Persistence |
|-----------|----------------|--------------|-------------|
| Embeddings | In Qdrant (local) | No | Until deleted |
| Content hashes | In .aioss ledger (local) | No | Permanent (immutable) |
| Parsed text | In content index (local) | No | Until garbage collected |
| Original file | On original filesystem | No | As-is |
| Thumbnails | In cache (local) | No | Until cache cleared |

### Can I Disable Content Analysis?

Yes. You can configure Kamelot to index files by metadata only (filename, path, dates, size) without reading file content:

```toml
[indexing]
content_analysis = false   # Index by metadata only
```

In this mode, Kamelot can still find files by filename, tag, and date — but not by semantic content.

## What Happens If I Lose My Keys?

### Scenario 1: Lost Master Key (Still Have Password)

If you have your encryption password but the master key file is corrupted:
```bash
kml keys recover --password
```

This re-derives the master key from your password and restores access.

### Scenario 2: Lost Password (Still Have Master Key)

If you have the master key file but forgot your password:
```bash
kml keys recover --master-key
```

This prompts for the master key file location and decrypts without password.

### Scenario 3: Lost Everything (No Key, No Password)

**Your encrypted files are permanently unrecoverable.** This is by design — without the encryption key, not even Kamelot Inc. can decrypt your files.

To prevent this:
1. **Back up your recovery key**: Print the recovery key phrase during `kml init` and store it in a safe (physical or password manager)
2. **Export your master key**: `kml keys export --output ~/backup/kamelot-master-key.enc`
3. **Back up your .aioss ledger**: The ledger contains all metadata needed to reconstruct the filesystem state, but the encrypted blobs require the key

### Recovery Key

During `kml init`, a human-readable recovery key phrase is displayed:

```
Your Kamelot recovery key phrase (store securely):
abandon-candy-frost-giraffe-hammer-igloo-jacket-kangaroo-lemon-mountain-neon-orange
```

You can use this to re-derive your master key at any time:
```bash
kml keys recover --phrase "abandon-candy-frost-giraffe-hammer-igloo-jacket-kangaroo-lemon-mountain-neon-orange"
```

## How Does Kamelot Protect Against Ransomware?

Kamelot's architecture provides inherent ransomware resistance through the **.aioss append-only ledger**.

### How It Works

1. **Append-only ledger**: Every file operation is recorded as an immutable entry. Ransomware cannot delete or modify past entries because the ledger is append-only and cryptographically chained.

2. **Content history**: When ransomware encrypts your files and writes them back, Kamelot records the encrypted versions as new entries — but the previous, unencrypted versions remain in the ledger's history.

3. **Rollback**: Using `kml rollback --minutes N`, you can revert all files to their state before the ransomware attack. The encrypted blobs are discarded, and the previous versions are restored.

### Ransomware Response Checklist

```
Step 1: Isolate — Disconnect from network (physical or software)
Step 2: Assess — kamelot doctor --ransomware
Step 3: Rollback — kml rollback --all --minutes 60
Step 4: Verify — kml fsck (checks file integrity)
Step 5: Restore — Re-index if needed
Step 6: Post-mortem — kayak report incident
```

### Zero Data Loss Guarantee

In controlled testing, Kamelot achieves **100% file recovery** from ransomware simulation. The immutable ledger ensures that every version of every file is available for rollback, provided the ledger itself has not been physically destroyed.

### Model Execution

- Models are loaded into a sandboxed process

### Model Supply Chain

| Risk | Mitigation |
|------|-----------|
| Malicious model weights | Checksum verification against official publisher |
| Model poisoning | Use only well-established models with provenance tracking |
| Model update hijacking | Signed model manifests, HTTPS downloads |

### Can the Model Exfiltrate Data?

No. The model runs locally with no network access. The only data it receives is your file content (for embedding generation). The only data it outputs is a vector embedding (high-dimensional numbers that cannot be reverse-engineered into file content).

## What About Memory Safety?

Kamelot is written in **Rust**, a memory-safe systems programming language. This means:

### What Rust Prevents
- **Buffer overflows**: The #1 cause of remote code execution vulnerabilities
- **Use-after-free**: The #2 cause of security vulnerabilities in C/C++
- **Null pointer dereferences**: Eliminated by the `Option<T>` type
- **Data races**: Prevented by the ownership system and Send/Sync traits
- **Iterator invalidation**: Caught at compile time
- **Format string vulnerabilities**: Eliminated by type-safe formatting macros

### What This Means for Security

| Language | Memory Safety | Known CVEs (2020-2025) | Kamelot Target |
|----------|-------------|----------------------|----------------|
| Rust | Guaranteed (safe code) | 0 (in standard library) | 0 memory safety CVEs |
| C | Manual | 500+ (kernel, system utils) | N/A |
| C++ | Manual (RAII helps) | 300+ (Chrome, system utils) | N/A |

### Unsafe Code Audit

Kamelot contains approximately 150 lines of `unsafe` Rust code, all in:
1. WinFSP FFI bindings (~80 lines)
2. libfuse3 FFI bindings (~50 lines)
3. Platform-specific memory operations (~20 lines)

All unsafe code is:
- Isolated in well-defined modules
- Wrapped in safe abstractions
- Heavily documented with safety invariants
- Tested with MIRI (Rust undefined behavior detection)
- Subject to additional code review

## How Are Encryption Keys Stored?

### Master Key Storage

The master encryption key is stored in `~/.kamelot/keys/master.key` with the following protections:

1. **OS-level encryption**: On supported platforms, the key file is encrypted using the OS keychain:
   - Linux: Secret Service API (libsecret) or encrypted file
   - Windows: DPAPI (Data Protection API)
   - macOS: Keychain

2. **Password protection (optional)**: If you set a password during `kml init`, the master key is encrypted with Argon2id key derivation:
   ```
   wrapped_key = Argon2id(password, salt, time=3, mem=64MB, parallelism=4)
   encrypted_master_key = XChaCha20-Poly1305(wrapped_key, master_key)
   ```

3. **File permissions**: The key file is created with 0600 permissions (owner read/write only)

### Key Export

```bash
# Export encrypted master key (for backup)
kml keys export --output backup/kamelot-key.enc

# Import master key (on new machine)
kml keys import --input backup/kamelot-key.enc

# Rotate master key
kml keys rotate
```

Key rotation re-encrypts all file entries with a new master key. It does not require re-encrypting the actual file blobs (since file keys are derived and only the derivation changes).

## Is the .aioss Ledger Tamper-Proof?

The .aioss ledger is **tamper-evident** (not tamper-proof — physical access can destroy it).

### Tamper Evidence Mechanism

Every entry in the .aioss ledger contains:
1. A SHA256 hash of the previous entry
2. A content hash of the file blob
3. A timestamp and sequence number

This creates a **hash chain**:

```
Entry 1: {hash: H1, content_hash: C1, prev_hash: 0}
Entry 2: {hash: H2, content_hash: C2, prev_hash: H1}
Entry 3: {hash: H3, content_hash: C3, prev_hash: H2}
...
```

### Verification

```bash
# Verify ledger integrity
kml ledger verify

# Output:
# ✓ Ledger entries: 42,569
# ✓ Hash chain: intact
# ✓ Content hashes: all verified
# ✗ Entry 12,340: content mismatch (file modified outside Kamelot)
# ✓ No tampering detected
```

### What Tampering Looks Like

| Attack | Detection | 
|--------|-----------|
| Modify an entry in place | Hash chain breaks (prev_hash mismatch) |
| Delete an entry | Gap in sequence numbers |
| Reorder entries | Sequence number discontinuity |
| Replace a file blob | Content hash mismatch |
| Insert a fake entry | Signature verification failure (v2.0+) |

## What Are the Encryption Overheads?

Encryption introduces minimal overhead in terms of performance and storage:

### Performance Overhead

| Operation | Without Encryption | With Encryption | Overhead |
|-----------|-------------------|-----------------|----------|
| File read (FUSE) | 50μs | 150μs | +100μs |
| File write (flat store) | 100μs | 300μs | +200μs |
| Decryption (per file open) | 0 | 50-200μs | 50-200μs |
| Encryption (per file index) | 0 | 100-500μs | 100-500μs |

### Storage Overhead

| Component | Without Encryption | With Encryption | Overhead |
|-----------|-------------------|-----------------|----------|
| Encrypted blob | File size + 0 | File size + 40 bytes | +40 bytes per file |
| Key storage | 0 | 32 bytes (master key) | Negligible |
| Total overhead | 0 | ~40 bytes/file | ~40 MB per 1M files |

The overhead is negligible for all practical purposes — less than 0.001% for typical file sizes.

### CPU Overhead

- AES-NI hardware acceleration is used on modern CPUs
- XChaCha20-Poly1305 is designed for efficient software implementation
- CPU impact: <1% during normal operation, <5% during heavy indexing

## How Are Encryption Keys Generated?

Keys are generated during `kml init` using the operating system's secure random number generator:

### Key Generation Process

```
1. OS random source (getrandom syscall on Linux, BCryptGenRandom on Windows,
   SecRandomCopyBytes on macOS)
   ↓
2. 256-bit master encryption key generated
   ↓
3. Master key encrypted with Argon2id (if password set)
   ↓
4. Recovery key phrase generated (BIP-39 style word list)
   ↓
5. All keys stored in ~/.kamelot/keys/ (encrypted)
```

### Key Quality Assurance

- OS random sources are cryptographically secure (CSPRNG)
- The Linux getrandom syscall blocks until the entropy pool is initialized
- Key generation is tested against NIST statistical randomness tests
- 256-bit keys provide 2^256 possible values — far beyond brute-force capability

### Key Generation Verification

```bash
# Verify key integrity
kml keys verify

# Check key strength
kml keys inspect

# Regenerate if compromised
kml keys rotate
```

## How Are File Permissions Handled in the Virtual Drive?

The Kamelot FUSE/WinFSP virtual drive respects filesystem permissions:

### Linux/macOS (FUSE)

| Permission | Behavior |
|------------|----------|
| Read (r) | File contents are accessible |
| Write (w) | Not supported in v0.1 (planned for v0.4) |
| Execute (x) | Files can be executed if executable bit set |
| Owner | Files appear owned by the user running the daemon |

### Windows (WinFSP)

| Permission | Behavior |
|------------|----------|
| Read | File contents are accessible |
| Write | Not supported in v0.1 (planned for v0.4) |
| Hidden files | Shown in the virtual drive (if indexed) |

### Permission Configuration

```toml
[fuse]
file_mode = 0o644    # Default file permissions (readable by all)
dir_mode = 0o755     # Default directory permissions
allow_other = false  # Only allow the mounting user to access
```

Setting `allow_other = true` allows other users on the system to access the virtual drive, but requires root/sudo for the initial FUSE mount.

## Can Malware Infect Files Through Kamelot's Virtual Drive?

The Kamelot FUSE/WinFSP virtual drive presents files to the operating system as regular files. This means:

### What Is Possible

- **Malware can read files** through the virtual drive (if it has OS-level access to the mount point)
- **Malware can write files** (when write support is enabled in future versions)
- **Malware can execute files** from the virtual drive (if they have execute permissions)

### What Is Mitigated

| Threat | Mitigation |
|--------|-----------|
| Ransomware encrypting files | Immutable ledger enables rollback |
| Malware modifying indexed files | Content hash verification detects changes |
| Malware accessing encrypted store | Flat store is encrypted at rest |
| Malware stealing embedding data | Vectors are not content (cannot reverse-engineer files) |
| Malware leveraging Kamelot for persistence | Daemon runs as user, not root |

### Best Practices

1. Keep your system antivirus active and scanning the Kamelot mount
2. Use OS-level permissions to restrict access to the virtual drive
3. Enable the immutable ledger for ransomware rollback protection
4. Regularly back up the ledger and flat store to external media

## Does Kamelot Support Hardware Security Modules (HSMs)?

Kamelot does not natively support HSMs, but can be configured to work with them:

### Current Limitations

- Keys are stored in software (encrypted on disk)
- No PKCS#11 or TPM integration
- No hardware-backed key generation

### Workaround: OS-Level HSM Integration

On Linux, you can use LUKS with a TPM-backed key for the Kamelot data directory:

```bash
# Create encrypted partition with TPM-backed LUKS
sudo cryptsetup luksFormat --type luks2 /dev/sdX
sudo systemd-cryptenroll --tpm2-device=auto /dev/sdX
sudo cryptsetup open /dev/sdX kamelot-secrets
sudo mkfs.ext4 /dev/mapper/kamelot-secrets
sudo mount /dev/mapper/kamelot-secrets ~/.kamelot
```

This ensures the entire Kamelot data directory (including keys) is encrypted with hardware-backed TPM protection.

### Planned HSM Support

Native HSM support (PKCS#11, TPM 2.0) is planned for v1.5+ for enterprise deployments requiring FIPS 140-2 compliance.

## What Is the Secure Deletion Policy?

Kamelot provides multiple levels of secure deletion:

### Delete Levels

| Level | Command | Behavior | Recoverable? |
|-------|---------|----------|--------------|
| Index removal | `kml delete --index-only` | Removes from search index only | Re-index restores |
| Ledger tombstone | `kml delete` | Marks as deleted, preserves history | Rollback restores |
| Purge | `kml delete --purge` | Removes from index, ledger, and flat store | Not recoverable |
| Secure wipe | `kml delete --secure-wipe` | Overwrites encrypted blob with zeros before deletion | Forensically unrecoverable |

### Secure Wipe Implementation

```bash
# 3-pass secure wipe (default)
kml delete --secure-wipe --passes 3 <inode>

# Single pass (fast, sufficient for SSDs)
kml delete --secure-wipe --passes 1 <inode>

# DoD standard (7 passes)
kml delete --secure-wipe --passes 7 <inode>
```

Note: On SSDs, secure deletion is limited by the SSD's internal wear-leveling and garbage collection. For truly sensitive data, full-disk encryption is recommended in addition to Kamelot's encryption.

## How Does Kamelot Handle Memory Scraping Attacks?

Memory scraping (reading sensitive data from RAM) is a concern for any application processing sensitive data. Kamelot implements several mitigations:

### Current Protections

1. **Sensitive data minimization**: Plaintext file contents are held in memory only during indexing, then released
2. **Encrypted storage**: The flat store holds only ciphertext (decrypted on-the-fly during access)
3. **Zero-copy where possible**: File contents are streamed through the parser without unnecessary copies
4. **Process isolation**: Qdrant and Ollama run as separate processes with limited memory sharing

### Limitations

- Decrypted file content in memory during active file access (via FUSE) is visible to root-level processes
- Encryption keys in memory are theoretically accessible to a root-level attacker

### Best Practices

1. Use full-disk encryption (FileVault, BitLocker, LUKS) as a second layer
2. Run Kamelot on a machine with restricted physical access
3. Use memory encryption (AMD SME / Intel TME) if available on the server
4. For highly sensitive data, consider running Kamelot on a dedicated, air-gapped machine

## How Does Kamelot Prevent Timing Attacks?

Timing attacks exploit variations in response time to infer sensitive information. Kamelot implements constant-time cryptographic operations:

### Protected Operations

| Operation | Timing Attack Vector | Mitigation |
|-----------|---------------------|------------|
| Key comparison | Timing reveals key bits | Constant-time comparison |
| Encryption | Timing reveals plaintext | XChaCha20 is constant-time |
| Hash comparison | Timing reveals hash bits | Constant-time comparison |
| Authentication | Timing reveals password length | Argon2id fixed time |

### Implementation

- Cryptographic primitives use `constant_time_eq` from the `subtle` crate
- All cryptographic operations run in fixed time regardless of input
- No early-exit optimization on authentication paths
