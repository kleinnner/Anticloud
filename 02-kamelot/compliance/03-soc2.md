## Overview

This document maps Kamelot features and controls to the SOC 2 Trust Services Criteria. It provides detailed guidance for SOC 2 auditors and compliance teams to assess Kamelot's readiness for SOC 2 Type II attestation.

Kamelot Enterprise deployments (Silver, Gold, and Platinum tiers) are designed to meet SOC 2 requirements. The system provides built-in controls for security, availability, processing integrity, confidentiality, and privacy.

### Trust Services Criteria Coverage Summary

| Category | Criteria Count | Kamelot Coverage | Evidence Type |
|----------|---------------|------------------|---------------|
| Security (CC6.x, CC7.x) | 9 | Full coverage | Audit logs, configuration, policies |
| Availability (A1.x) | 3 | Full coverage | Uptime reports, DR documentation |
| Processing Integrity (PI1.x) | 3 | Full coverage | Processing logs, validation reports |
| Confidentiality (C1.x) | 3 | Full coverage | Classification config, access logs |
| Privacy (P1.x-P6.x) | 9 | Partial (customer-dependent) | Privacy controls, consent records |

### Scope of SOC 2 Assessment

The SOC 2 assessment covers the Kamelot semantic vector file system as deployed on customer-controlled infrastructure. The boundaries of the assessment include:

- Kamelot daemon (FUSE mount, gRPC API, management UI).
- Qdrant vector database (storage and similarity search).
- Ollama embedding engine (Qwen 2 VL Q4 model).
- Redis cache layer (optional, session management).
- Encrypted flat object store (file content storage).
- .aioss immutable audit ledger (audit logging).
- Backup and disaster recovery systems.

Excluded from scope: customer application code, customer network infrastructure, end-user devices, third-party identity providers (LDAP/SAML/OIDC servers are assessed separately).

## Trust Services Criteria

### CC6.1: Logical and Physical Access

Kamelot implements logical access controls through authentication, authorization, and encryption mechanisms.

**Kamelot Implementation**:
- Authentication: LDAP, SAML 2.0, OIDC, or local accounts with MFA support.
- Authorization: Hierarchical RBAC with four base roles and custom roles.
- Encryption: TLS 1.3 for all network communication, AES-256-GCM for data at rest.
- Session management: JWT-based sessions with configurable TTL and idle timeout.

**Audit Evidence**:
- Authentication configuration files.
- RBAC policy definitions.
- TLS certificate inventory.
- Session configuration documentation.

### CC6.2: User Access Authorization

Kamelot authorizes all user access based on authenticated identity and assigned permissions.

**Kamelot Implementation**:
- Every API request includes authentication credentials (JWT or API token).
- The RBAC engine resolves permissions for every operation.
- The ABAC policy engine applies additional attribute-based restrictions.
- All authorization decisions are logged in the audit ledger.

**Audit Evidence**:
- User list with role assignments.
- Group-to-role mapping configuration.
- Authorization audit logs showing allow/deny decisions.
- RBAC compliance reports.

### CC6.3: Access Authorization

Kamelot restricts access to resources based on role, ownership, sharing, and policy.

**Kamelot Implementation**:
- File access: owner-only by default, shareable explicitly.
- Search results: filtered by user permissions.
- Administrative functions: restricted to admin role.
- Audit access: restricted to admin and auditor roles.

**Audit Evidence**:
- File permission audit logs.
- Sharing records.
- Policy evaluation logs.
- Access denied event reports.

### CC6.4: Security of Data in Transit

All data transmitted over networks is encrypted using TLS 1.3.

**Kamelot Implementation**:
- TLS 1.3 for HTTPS API and management UI.
- TLS 1.3 for gRPC communication with Qdrant.
- TLS 1.3 for HTTP communication with Ollama.
- mTLS for inter-service authentication in production deployments.
- Certificate pinning for critical service connections.

**Audit Evidence**:
- TLS configuration files.
- Certificate authority certificates.
- Certificate inventory and expiration tracking.
- Network security scan results.

### CC6.5: Security of Data at Rest

All stored data is encrypted using AES-256-GCM.

**Kamelot Implementation**:
- File content: encrypted with unique per-file key, wrapped with master key.
- Vector embeddings: stored in Qdrant with optional encryption.
- Audit ledger: content is not encrypted but is cryptographically chained and signed.
- Configuration files: sensitive values are encrypted.
- Key management: automatic rotation, HSM support, access logging.

**Audit Evidence**:
- Encryption configuration files.
- Key management policy documentation.
- Key rotation audit logs.
- Data encryption verification reports.

### CC6.6: Security of Data Disposal

Kamelot provides secure data disposal through retention policies and compliance deletion workflows.

**Kamelot Implementation**:
- Retention policies define disposal timelines for each data type.
- Automatic pruning removes expired vectors and objects.
- Compliance deletions remove all data for a specific user or classification.
- Deleted data is securely purged (zeroed before freeing).
- Backup copies are flagged for purging on the next cycle.

**Audit Evidence**:
- Retention policy configuration.
- Pruning audit logs.
- Compliance deletion records.
- Data disposal verification reports.

### Uptime Measurement

Uptime is measured using the following formula:

`Monthly Uptime % = ((Total Minutes in Month - Downtime Minutes) / Total Minutes in Month) * 100`

Downtime is defined as any period when the Kamelot API is not responding to health check requests. Planned maintenance with 14-day notice is excluded from downtime calculations.

## Confidentiality: Data Classification

Kamelot supports data classification to enforce confidentiality controls.

### Classification Levels

| Level | Description | Access Restriction | Default Retention |
|-------|-------------|-------------------|-------------------|
| Public | No restrictions | Any authenticated user | 90 days |
| Internal | Company-internal only | All users in the organization | 365 days |
| Confidential | Specific authorized users | Role + department-based | 1095 days |
| Restricted | Named individuals only | Explicit user list | 2555 days |
| Regulatory | Compliance-controlled | Role + policy-based | 2555+ days |

### Classification Enforcement

Data classification is enforced by the policy engine at every access attempt. When a user requests access to a classified file:

1. The RBAC engine checks if the user's role permits access to the classification level.
2. The ABAC policy engine evaluates department, team, and attribute-based policies.
3. If all checks pass, access is granted and logged.
4. If any check fails, access is denied, the event is logged, and an alert may be triggered.

## Confidentiality: Encryption Controls

Encryption serves as the primary technical control for data confidentiality.

### Key Management Architecture

Kamelot uses envelope encryption:

1. Each file is assigned a unique 256-bit data key (DEK).
2. The DEK encrypts the file content using AES-256-GCM.
3. The DEK is wrapped (encrypted) using a master key (KEK).
4. The wrapped DEK is stored alongside the encrypted file.
5. The KEK is stored in an HSM or encrypted key file.
6. When reading a file, the KEK decrypts the DEK, which decrypts the file.

### Key Rotation

| Key Type | Rotation Period | Method | Impact |
|----------|----------------|--------|--------|
| Data key (DEK) | Per-file | Each new file gets a new DEK | None |
| Master key (KEK) | 90 days (configurable) | New KEK wraps existing DEKs | Background re-encryption |
| TLS certificate | 365 days | Certificate re-issuance | Connection restart |
| Signing key | 365 days | New key pair generated | Ledger re-signing |

## Processing Integrity

Kamelot ensures processing integrity through validation, error handling, and verification mechanisms.

### Data Validation

- All file writes compute SHA-256 checksum before and after encryption.
- Checksums are verified on every read operation.
- Vector embeddings are validated for dimensional consistency (768 dimensions for Qwen 2 VL).
- Metadata fields are validated against defined schemas.
- File size limits are enforced at the API level.

### Error Handling

- All processing errors are logged with full context (error type, stack trace, affected resource).
- Failed write operations are rolled back atomically (all-or-nothing).
- The system enters a safe state on critical errors (e.g., encryption key unavailable).
- Administrators are notified of processing failures via the configured alert channel.
- Error events are tracked and reviewed in weekly operations meetings.

## Expanded Section: Detailed SOC 2 Implementation

### CC6.7: Security of Data Disposal

Kamelot implements secure data disposal procedures that ensure data is irrecoverable after deletion.

**Disposal Methods by Data Type**:
- Vector embeddings: Qdrant hard deletion with HNSW graph rebuild.
- File objects: Cryptographic erasure (zeroing) followed by file system TRIM.
- Metadata: Database record deletion with index rebuild.
- Backup media: Overwritten on next backup cycle.
- Logs: Secure deletion with cryptographic chain verification.

**Disposal Verification**: After disposal, the system verifies that data is no longer accessible through any API or storage path. Verification results are logged in the audit ledger.

### CC7.3: Security Incident Reporting

Kamelot includes automated security incident reporting capabilities:

1. **Detection**: Automated monitoring detects potential incidents (unusual access patterns, failed authentication spikes, tamper detection).
2. **Triage**: Incidents are automatically categorized by severity (critical, high, medium, low).
3. **Notification**: Stakeholders are notified via configured channels (email, Slack, PagerDuty, SIEM).
4. **Documentation**: Incidents are documented in the incident ledger with timestamps, affected resources, and response actions.
5. **Response**: Playbook-driven response procedures guide the incident response team.
6. **Recovery**: Automated recovery procedures restore normal operation.
7. **Post-mortem**: Incident review and lessons learned are documented.

### A1.2: Availability - Disaster Recovery

**Silver Tier DR**:

**Gold Tier DR**:

Kamelot enforces access restrictions through multiple layers:

4. **Audit Logging**: Enable comprehensive audit logging for all operations. Configure SIEM integration for real-time monitoring.
5. **Vulnerability Management**: Regularly scan container images for vulnerabilities. Apply security patches within the SLA timeframe.

### Troubleshooting Guide

#### Symptom: Feature not working as expected

1. Verify the feature is enabled in the configuration file.
2. Check the logs for error messages: `kamelotctl logs --level error`.
3. Validate the configuration: `kamelotctl config validate`.
4. Restart the service: `kamelotctl restart`.
5. If the issue persists, contact support with the diagnostic output: `kamelotctl diag collect`.

#### Symptom: Performance degradation

1. Check resource usage: `kamelotctl status`.
2. Review performance metrics in Grafana or Prometheus.
3. Increase resource allocation if necessary.
4. Optimize configuration parameters (batch size, thread count, cache settings).
5. Consider scaling horizontally by adding more nodes.

#### Symptom: Integration failure

1. Verify connectivity between services: `kamelotctl diag network`.
2. Check authentication credentials and tokens.
3. Verify TLS certificate validity.
4. Check firewall rules and network segmentation.
5. Review integration logs for specific error messages.
