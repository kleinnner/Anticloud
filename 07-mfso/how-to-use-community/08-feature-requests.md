## Introduction

MF+SO is built for its users. Feature requests from the community directly shape the product roadmap. Every feature in MF+SO started as someone's idea — from the duress PIN to the .aioss ledger.

This guide explains how to submit feature requests, how they're evaluated, and how you can influence the development direction.

### The Product Development Cycle

```
Your Idea → Feature Request → Community Voting → Team Review → 
  Roadmap → Development → Beta Testing → Release → Your Feedback
```

## How Feature Requests Work

### The Process

1. **Submit**: You submit a feature request through one of our channels
2. **Review**: The product team reviews within 2 weeks
3. **Status assigned**: One of:
   - **Under Review**: Being evaluated
   - **Planned**: Scheduled for development
   - **In Progress**: Being built
   - **Completed**: Released
   - **Declined**: Not moving forward (with explanation)
   - **Duplicate**: Already requested (link to original)
4. **Community voting**: Others vote and comment
5. **Roadmap**: High-demand features enter the roadmap
6. **Development**: Built by the team or community contributors
7. **Release**: Shipped to all users

### What Makes a Good Feature Request

A good feature request includes:

1. **Problem Statement**: What problem does this solve?
2. **User Story**: "As a [type of user], I want [goal] so that [reason]"
3. **Proposed Solution**: What would you like to see?
4. **Alternatives Considered**: What else have you tried?
5. **Use Cases**: When would you use this?
6. **Priority**: How important is this to you?

### What Happens After Submission

| Status | Meaning | Timeline |
|--------|---------|----------|
| New | Just submitted | Initial review within 2 weeks |
| Under Review | Team is evaluating | 2-4 weeks |
| More Info Needed | Team needs clarification | Until you respond |
| Planned | Approved for development | Scheduled on roadmap |
| In Progress | Being built | Next release cycle |
| Completed | Released | Current or upcoming release |
| Declined | Will not be built | Explanation provided |
| Duplicate | Already has a request | Linked to original |

## Submitting a Feature Request

### Method 1: Community Forum (Recommended)

1. Go to https://community.mfso.app
2. Navigate to "Feature Requests" category
3. Click "New Post"
4. Use the feature request template:

```
## Problem Statement
[What problem are you trying to solve?]

## User Story
As a [type of user],
I want [feature or capability]
so that [benefit or value].

## Proposed Solution
[Describe what you'd like to see]

## Alternatives Considered
[What else have you tried or considered?]

## Use Cases
[When and how would you use this?]

## Priority
[Critical / High / Medium / Low]

## Additional Context
[Screenshots, mockups, or examples]
```

5. Submit the post
6. Engage with comments and questions

### Method 2: In-App Feedback

1. Go to Settings → Feedback
2. Select "Feature Request"
3. Describe your request
4. Optionally attach screenshots
5. Submit

### Method 3: GitHub Issues

For technical feature requests or developer-facing features:

1. Go to https://github.com/mfso/mfso/issues
2. Click "New Issue"
3. Choose "Feature Request" template
4. Fill in the details
5. Submit

### Method 4: Discord

1. Go to the `#feature-requests` channel
2. Use the `/feature-request` command
3. Fill in the form
4. The request is automatically posted for discussion

## Voting on Features

### How Voting Works

Community voting helps us understand what features matter most:

- **Upvote**: Shows support for a feature
- **Comment**: Adds context, use cases, or suggestions
- **Share**: Spread the word about important features

### Voting Guidelines

- Vote for features you would actually use
- Only upvote once per feature (duplicate votes are removed)
- Comment with constructive feedback
- Don't create duplicate requests to boost visibility

### Voting Weight

| User Type | Vote Weight | Why |
|-----------|-------------|-----|
| Free user | 1 vote | Community voice |
| Premium user | 2 votes | Invested user perspective |
| Enterprise user | 3 votes | Organizational needs |
| Contributor | 2 votes | Active community involvement |
| Beta tester | 2 votes | Testing experience |

### What Votes Mean

- **50+ votes**: Feature enters formal review
- **100+ votes**: Feature prioritized for next roadmap
- **200+ votes**: Feature becomes a top priority
- **500+ votes**: Feature likely to be fast-tracked

### Viewing Top Requests

See the most popular feature requests at:
- https://community.mfso.app/feature-requests/top
- In-app: Help → Feature Requests → Top

## Public Roadmap

### Accessing the Roadmap

- **Web**: https://roadmap.mfso.app
- **In-app**: Help → Roadmap

### Roadmap Categories

| Category | Meaning |
|----------|---------|
| 🎯 Now | Currently being worked on |
| 📋 Next | Planned for the next quarter |
| 🔭 Later | Being considered for future |
| ✅ Done | Recently completed |

### What's on the Roadmap

The roadmap shows:
- **Feature name and brief description**
- **Status**: Now / Next / Later / Done
- **Progress**: For "Now" items: 0-100%
- **Expected release**: Quarter and year
- **Upvotes**: Community interest level
- **Contributors**: If community members are working on it

### Roadmap Updates

- Updated at the start of each month
- Major changes communicated in release notes
- Quarterly roadmap review with community input

### Feature Status Definitions

| Status | Description | Can It Change? |
|--------|-------------|----------------|
| Investigating | We're exploring the feasibility | Yes — may proceed or decline |
| Prototyping | We're building a prototype | Usually proceeds to development |
| In Development | Engineering is actively building | Proceeds to release |
| Beta Testing | Available in beta | Proceeds to general release |
| Released | Available to all users | Completed |
| Deferred | Postponed to a later date | May be revisited |
| Declined | Will not be pursued | Unlikely unless new context |

## Development Prioritization

### How Features Are Prioritized

Features are evaluated on several factors:

| Factor | Weight | Description |
|--------|--------|-------------|
| Community demand | 30% | Number and quality of upvotes |
| User impact | 25% | How many users benefit |
| Strategic alignment | 20% | Fits product vision |
| Development effort | 10% | Time and resources required |
| Technical feasibility | 10% | Can we build it? |
| Business value | 5% | Impact on sustainability |

### Priority Matrix

| Community Demand | Development Effort | Result |
|-----------------|-------------------|--------|
| High | Low | Fast-track (quick wins) |
| High | High | Major investment (roadmap) |
| Low | Low | Niche features (when possible) |
| Low | High | Unlikely (won't do) |

### Who Decides?

- **Product Team**: Evaluates and prioritizes
- **Engineering Team**: Assesses technical feasibility
- **Community**: Provides input through voting and feedback
- **Security Team**: Reviews security implications

### Transparency

When a feature is declined, we explain why:
- **Low demand**: Not enough community interest
- **Too complex**: Development effort too high for impact
- **Security concern**: Introduces unacceptable risk
- **Scope creep**: Doesn't fit the product vision
- **Privacy impact**: Conflicts with zero-knowledge principles

   - **Desktop**: Windows, macOS, Linux
   - **Mobile**: iOS (TestFlight), Android (Play Store beta)

### Beta Expectations

- **Early access**: You get features before general release
- **Stability**: Beta software is stable but may have issues
- **Feedback**: Your input shapes the final release
- **Frequency**: Beta updates more often (weekly vs monthly)

### Providing Beta Feedback

1. Use the in-app feedback button (shake gesture on mobile)
2. Discord `#beta` channel
3. GitHub issues (for bugs)
4. Beta feedback form (emailed with each beta)

### Beta Rewards

Beta testers receive:
- **Early access** to new features
- **Special Discord role**: Beta Tester
- **Direct line** to the development team
- **Influence**: Your feedback directly shapes releases

## Providing Feedback

### Types of Feedback

| Type | Best Channel | Impact |
|------|-------------|--------|
| Feature requests | Community forum | Roadmap influence |
| Bug reports | GitHub + in-app | Product quality |
| UX feedback | In-app + forum | Design improvements |
| Performance issues | GitHub + email | Technical improvements |
| Documentation feedback | GitHub docs repo | Better guides |
| Security concerns | security@mfso.app | Safety improvements |

### Structured Feedback

Good feedback includes:

1. **Context**: What were you doing?
2. **Expectation**: What did you expect to happen?
3. **Reality**: What actually happened?
4. **Suggestion**: What would be better?
5. **Importance**: How much does this matter to you?

### Feedback Principles

- **Specific**: "The search bar should support regex" vs "Search is bad"
- **Constructive**: "Adding a dark mode would help with eye strain" vs "This is hard to look at"
- **Actionable**: "Add a confirmation dialog before deleting entries" vs "This is dangerous"
- **Respectful**: Be kind — we're all trying to make a great product

### What We Do With Feedback

| Feedback Volume | Action |
|-----------------|--------|
| 1-5 similar requests | Track, monitor for trends |
| 5-20 similar requests | Discuss in product review |
| 20-100 similar requests | Prioritize for roadmap |
| 100+ similar requests | Fast-track development |

### Acknowledgment

- You'll receive an acknowledgment within 48 hours
- Status updates as the request is evaluated
- Notification when the feature is released (if implemented)

## FAQ

### General

**Q: How long does it take for a feature to be implemented?**
A: It varies. Simple features can ship in weeks; complex features may take months. High-demand features are prioritized.

**Q: Can I pay for a feature to be developed?**
A: Enterprise customers can sponsor feature development. For community features, we follow the public roadmap.

**Q: How do I know if my feature request was received?**
A: You'll get an acknowledgment within 48 hours (forum, GitHub, or email).

**Q: Can I request a feature for someone else?**
A: Yes, but please explain the use case and who it helps.

### Voting

**Q: Can I change my vote?**
A: Yes, you can remove your upvote at any time.

**Q: Can I vote multiple times?**
A: No. One vote per feature request per user (voting weight may differ by user type).

**Q: Do downvotes exist?**
A: No. We only track positive interest. If you disagree with a feature, explain why in comments.

### Implementation

**Q: Can I implement a feature myself?**
A: Yes! MF+SO is open source. See our contributing guidelines on GitHub.

**Q: Will the team help me implement my feature?**
A: Yes. We review pull requests and provide guidance. Join our Discord `#development` channel.

**Q: What if my feature request conflicts with the product vision?**
A: We'll explain why it doesn't fit and suggest alternatives.

### Enterprise

**Q: Can enterprise customers request custom features?**
A: Yes. Enterprise customers have a dedicated account manager who handles feature requests.

**Q: Are enterprise feature requests prioritized over community requests?**
A: Enterprise requests are evaluated separately. Some features only make sense for enterprise deployments.

**Q: Can enterprise features become community features?**
A: Yes Many features started as enterprise requests before being released to all users.
