## 1. Executive Summary

MF+SO is designed as a sovereign identity and authentication vault that operates on a local-first, zero-knowledge architecture. As such, it presents a unique audit profile for SOC 2 certification. Unlike cloud-dependent SaaS products, MF+SO processes authentication events entirely on the user's device. The MF+SO relay server acts solely as a blind packet relay with zero visibility into the contents of the data being transmitted.

This document outlines the comprehensive controls and evidence mapping required to achieve and maintain SOC 2 Type II certification. The scope covers the MF+SO relay infrastructure, the open-source codebase, the build and distribution pipeline, and the organizational policies governing development and operations.

### 1.1 Design Philosophy and Audit Implications

SOC 2 audits traditionally assume a centralized service provider model where the audited entity stores, processes, or transmits customer data on its own infrastructure. MF+SO's architecture inverts this assumption. The following architectural decisions fundamentally shape our compliance approach:

- **Local-first processing**: All cryptographic operations occur on the user's device. The relay server never possesses plaintext keys, credentials, or session data.
- **Zero-knowledge relay**: The server cannot distinguish between a legitimate authentication request and random noise. Packet payloads are encrypted end-to-end between MF+SO instances.
- **No user database**: MF+SO does not maintain a registry of users, accounts, or authentication events. The .aioss chain is user-held and user-verified.
- **Open source transparency**: The entire codebase is publicly auditable. Builds are reproducible, and binary attestations are signed.

These design choices reduce the attack surface dramatically but also require a reimagined approach to SOC 2 controls. Traditional controls around data center access, database encryption, and user access reviews must be adapted for an infrastructure that is intentionally minimalist.

### 1.2 Certification Scope

The SOC 2 examination will cover the following system components:

| Component | Description | In Scope |
|-----------|-------------|----------|
| MF+SO Relay Server | Packet relay infrastructure (STUN/TURN-like) | Yes |
| MF+SO Client Application | Progressive Web App (PWA) | No (user-controlled) |
| Build & Distribution Pipeline | CI/CD, code signing, artifact storage | Yes |
| .aioss Chain Specification | Protocol definition | No (standard) |
| Public Key Infrastructure | Key generation, distribution | Yes |
| Support & Maintenance | Incident response, updates | Yes |

The client application is explicitly **out of scope** because it runs in the user's browser under their control. However, the build pipeline that produces the client application is in scope to ensure supply chain integrity.

## 2. Scope of Certification

### 2.1 Organizational Context

MF+SO is developed and maintained by Lois-Kleinner in partnership with 0-1.gg. The organizational structure is distributed-first with team members across multiple jurisdictions. The SOC 2 report will cover the entity "Lois-Kleinner" as the service organization.

### 2.2 System Boundary

The system boundary encompasses:

1. **Infrastructure Layer**: Cloud relay servers, DNS, CDN for PWA distribution
2. **Application Layer**: Relay server software, build tools, signing infrastructure
3. **Data Layer**: Ephemeral relay state (no persistent storage beyond configuration)
4. **Support Layer**: Issue tracking, security advisories, update distribution

### 2.3 Subservice Organizations

| Subservice | Service Provided | Oversight Method |
|------------|-----------------|------------------|
| Cloud Provider (Tier 2) | Virtual machines, networking | SOC 2 report review, contractual safeguards |
| CDN Provider (Tier 2) | PWA distribution | SOC 2 report review |
| Code Repository Host | Source code management | Organizational controls, access reviews |

### 2.4 Complementary User Entity Controls

Users of MF+SO are responsible for:

- Securing their devices with appropriate access controls
- Safeguarding their .aioss chain backup phrases
- Keeping their browsers and operating systems updated
- Using strong device passwords or biometric authentication
- Verifying the integrity of downloaded MF+SO builds

SOC 2 controls are designed with the assumption that user entities implement these complementary controls. The service organization's controls cannot fully protect against inadequate user entity controls.

3. **Containment, Eradication & Recovery**: Affected systems are isolated, root cause is removed, services are restored
4. **Post-Incident Activity**: Lessons learned are documented, controls are improved

|----------|--------------|-----------------|
| Critical | 15 minutes | 4 hours |
| High | 1 hour | 24 hours |
| Medium | 4 hours | 72 hours |

## 11. System Description

### 11.1 System Overview

MF+SO is a sovereign identity and authentication vault that provides multi-factor authentication without reliance on third-party infrastructure. The system consists of the following major components:

#### 11.1.1 MF+SO Client Application

The client application is a Progressive Web App (PWA) that runs entirely in the user's browser. It performs all cryptographic operations locally, maintains the .aioss chain, and communicates with other MF+SO instances via the relay server.

**Technical Stack**:
- Frontend: TypeScript, WebAssembly (Rust-compiled crypto)
- Storage: IndexedDB, Web Crypto API
- Communication: WebRTC, WebSocket
- Chain: .aioss (SHA3-256 linked hash chain)

#### 11.1.2 MF+SO Relay Server

The relay server facilitates peer-to-peer communication between MF+SO instances. It is intentionally stateless and has zero knowledge of the contents of relayed packets.

**Technical Stack**:
- Server: Rust (Actix Web)
- Networking: WebRTC, WebSocket, STUN/TURN
- Telemetry: Anonymized metrics only
- Deployment: Multi-region, containerized

#### 11.1.3 Build and Distribution Pipeline

The build pipeline compiles the MF+SO client application from source, signs the artifacts, and distributes them via CDN and PWA update mechanisms.

**Components**:
- CI/CD: GitHub Actions
- Code signing: Signify/Signify (Minisign-compatible)
- Artifact storage: Sigstore for transparency
- Distribution: CDN with integrity checksums

### 11.2 Boundaries

The system boundaries are defined by the interaction between in-scope and out-of-scope components:

