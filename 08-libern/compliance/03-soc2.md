# SOC 2 Trust Services Criteria

## 1. Overview

SOC 2 (Service Organization Control 2) is an auditing framework developed by the
American Institute of CPAs (AICPA) that evaluates service organizations against
five Trust Services Criteria: Security, Availability, Processing Integrity,
Confidentiality, and Privacy. A SOC 2 report demonstrates that a service
organization has appropriate controls in place to protect customer data.

Libern's architecture — local-first, P2P, no cloud dependency — creates a unique
compliance posture. Because Libern is software deployed on customer-owned
infrastructure rather than a cloud service, the SOC 2 assessment applies primarily
to the organization deploying Libern, with Libern providing the technical controls
necessary to meet the criteria.

This document maps Libern's features and capabilities to each of the five Trust
Services Criteria and provides guidance for organizations seeking SOC 2 compliance
for their Libern deployment.

## 2. Trust Services Criteria Mapping

### 2.1 Security

The Security criterion refers to the protection of system resources against
unauthorized access, use, or modification.

#### 2.1.1 CC6.1 — Logical and Physical Access Controls

Libern provides robust access controls at multiple layers:

- **Identity-based access:** Each user is identified by an Ed25519 cryptographic
  key pair. No passwords, no shared secrets.
- **Server authorization:** Peers must be explicitly authorized by the server
  administrator to join a server.
- **Channel-level permissions:** Role-based access control (RBAC) determines
  which users can read, write, or administer specific channels.
- **Physical access:** Libern runs on customer-controlled hardware. Physical
  security is managed by the deploying organization.

#### 2.1.2 CC6.3 — Authorization and Approval

- All access to server data is mediated by cryptographic verification.
- Administrative actions (adding peers, modifying roles, deleting channels)
  require the server creator's authorization.
- The .aioss ledger records all authorization changes with cryptographic proof.

#### 2.1.3 CC6.6 — Transmission Security

- All P2P communication is encrypted using TLS-like channels.
- mDNS discovery operates within the local network segment only.
- No data is transmitted to any third-party infrastructure.

#### 2.1.4 CC7.1 — Identification and Authentication

- Ed25519 public keys serve as unique identifiers.
- Authentication is cryptographic: possession of the corresponding private key
  proves identity.
- Key compromise detection is supported through audit log analysis.

#### 2.1.5 CC7.2 — System Monitoring

- The .aioss ledger provides comprehensive monitoring of all system operations.
- Every message, file transfer, and configuration change is recorded.
- The hash chain enables detection of tampering or unauthorized modifications.
- Monitoring dashboards can be built on top of the SQLite database or .aioss
  export.

#### 2.1.6 CC7.3 — Incident Response

Libern supports incident response through:

- Cryptographic audit trails that provide forensic evidence
- Rapid isolation of compromised peers by removing them from the server
- Key rotation procedures that invalidate compromised credentials
- Data recovery capabilities through .aioss exports and CRDT state reconciliation

### 2.2 Availability

The Availability criterion refers to the accessibility of the system and its
services as committed or agreed.

#### 2.2.1 A8.1 — Available for Operation and Use

Libern's P2P architecture provides inherent availability advantages:

- **No single point of failure:** There is no central server to go down. Each
  peer maintains its own complete copy of the server state.
- **Offline resilience:** Libern is offline-first. Users can work without any
  network connectivity and synchronize changes when connectivity is restored.
- **No cloud dependency:** The system does not depend on any external service
  for core functionality. Even local AI inference runs on the local device.
- **CRDT convergence:** When connectivity is restored after an outage, peers
  automatically converge to a consistent state using CRDT merge logic.

#### 2.2.2 A8.2 — Monitoring of Availability

- Availability can be monitored through peer connection status indicators.
- The .aioss ledger records connectivity and synchronization events.
- Organizations can implement additional monitoring by inspecting peer heartbeat
  messages and sync intervals.

#### 2.2.3 A8.3 — Handling of Data Storage and Processing Capacity

- Storage capacity is determined by the local device's disk space.
- The SQLite database and .aioss ledger are designed for efficient storage.
- Large files can be managed through attachment policies.
- Processing capacity scales with the local CPU and GPU resources.

### 2.3 Processing Integrity

The Processing Integrity criterion refers to the completeness, accuracy,
timeliness, and authorization of system processing.

#### 2.3.1 PI1.1 — Processing Is Complete and Accurate

- **CRDT guarantee:** Conflict-Free Replicated Data Types ensure that all peers
  converge to the same state, even with concurrent edits.
- **Cryptographic verification:** Every operation is signed and verified.
  Invalid operations are rejected.
- **Hash chain integrity:** The .aioss ledger provides a complete, unbroken
  record of all processing.

#### 2.3.2 PI1.2 — Processing Is Authorized

- All processing is user-initiated. No background or automated processing occurs
  without user intent.
- Operations that affect server state require appropriate authorization (owner,
  admin, or member with relevant permissions).
- The authorization chain is recorded in the .aioss ledger.

#### 2.3.3 PI1.3 — Processing Is Timely

- Libern processes operations in real-time for online peers.
- Offline operations are queued and synchronized when connectivity is available.
- CRDT merge ensures that all operations are eventually processed in a
  consistent order.

### 2.4 Confidentiality

The Confidentiality criterion refers to the protection of information designated
as confidential.

#### 2.4.1 C1.1 — Confidential Information Is Protected

- All data is stored locally on the user's device. It is not transmitted to any
  external service.
- P2P communication is encrypted in transit.
- Access to confidential information is controlled by cryptographic identity
  verification and role-based permissions.
- The open-source codebase allows organizations to verify confidentiality
  protections.

#### 2.4.2 C1.2 — Confidential Information Is Disposed of

- Local deletion: Data can be deleted immediately by removing the local database
  or individual records.
- .aioss export for archival: Confidential information can be exported and then
  securely deleted.
- Secure deletion: Organizations should use OS-level secure deletion tools
  (shred, srm) for sensitive data removal.

#### 2.4.3 C1.3 — Confidential Information Is Not Improperly Disclosed

- No automatic data sharing: Data is only shared with explicitly authorized peers.
- No telemetry: Libern does not collect or transmit usage statistics, error
  reports, or analytics.
- No third-party access: The architecture has no backdoors, no remote access
  capabilities, and no data processing agreements with external entities.

### 2.5 Privacy

The Privacy criterion refers to the collection, use, retention, disclosure, and
disposal of personal information.

#### 2.5.1 P1.1 — Privacy Notice

Libern supports privacy notice compliance by:

- Being fully transparent through open source code
- Not collecting any personal information beyond what the user explicitly creates
- Providing clear documentation of all data flows

#### 2.5.2 P1.2 — Choice and Consent

- Users choose what data to create and share.
- Consent is managed through peer authorization — no data is shared without
  explicit acceptance.
- Users can withdraw consent at any time by deleting data or removing peers.

#### 2.5.3 P1.3 — Collection Is Limited

- Libern collects only the data that users explicitly create: messages, files,
  voice data, and whiteboard canvas data.
- No metadata collection, no browsing history, no usage analytics.
- Data minimization is baked into the architecture.

#### 2.5.4 P1.4 — Use Is Limited

- Data is used only for the purposes of local collaboration and communication.
- No secondary uses, no data mining, no training of AI models on user data.
- Local AI inference uses the local model only and does not transmit any
  prompts or responses to external services.

#### 2.5.5 P1.5 — Retention Is Limited

- Users control data retention. They can delete individual records, channels,
  or entire servers at any time.
- No automatic retention policies are enforced by the software.
- .aioss export allows archival before deletion.

#### 2.5.6 P1.6 — Disposal Is Timely

- Deletion is immediate when the user takes action.
- CRDT tombstones ensure that deleted content is not accessible through normal
  application interfaces.
- Physical data removal from the storage medium depends on the OS-level delete
  behavior.

#### 2.5.7 P1.7 — Disclosure to Third Parties

- Libern does not disclose personal information to any third party.
- There is no mechanism for third-party data access.
- The only data sharing is between explicitly authorized peers.

#### 2.5.8 P1.8 — Privacy Reviews

- The open-source codebase can be audited by any privacy reviewer.
- The .aioss ledger provides a complete record of all data processing.
- Organizations can conduct privacy reviews by inspecting code, configuration,
  and audit logs.

## 3. SOC 2 Type I vs. Type II

### 3.1 Type I Readiness

Libern's architecture provides a strong foundation for SOC 2 Type I reporting
(design suitability). The controls described in this document are designed into
the software and are active from the moment of deployment.

### 3.2 Type II Considerations

For SOC 2 Type II (operating effectiveness), organizations deploying Libern
should:

1. Establish policies and procedures for monitoring the controls described above
2. Document the operating effectiveness of controls over the audit period
3. Review .aioss audit logs regularly to verify control operation
4. Test incident response procedures using Libern's capabilities
5. Conduct periodic access reviews of authorized peers

## 6. Implementation Guidance for SOC 2 Readiness

### 6.1 Pre-Audit Preparation

Organizations preparing for a SOC 2 audit should:

1. **Map controls:** Use Section 2 of this document as the control mapping
   baseline
2. **Document policies:** Create policies for each applicable control area
3. **Configure auditing:** Ensure .aioss ledger is configured for appropriate
   retention
4. **Establish monitoring:** Set up procedures for regular control review
5. **Train personnel:** Ensure administrators and users understand their roles
6. **Gather evidence:** Begin collecting evidence of control operation

### 6.2 Evidence Collection

For SOC 2 audit, collect evidence of:

| Control | Evidence Source | Collection Method |

| Access control | .aioss authorization records | Export ledger |

| Availability | Peer uptime logs, incident records | Monitoring tools |
| Processing integrity | CRDT convergence records | System logs |
| Incident response | Incident documentation, .aioss forensic data | Incident tracker |

### 6.3 Common Audit Findings and Remediation

| Finding | Likelihood with Libern | Remediation |

## 7. Continuous Monitoring

### 7.1 Automated Monitoring with .aioss

The .aioss ledger enables continuous monitoring of controls:

```bash
# Verify hash chain integrity (automated, daily)
libern --verify-ledger /path/to/ledger.aioss

# Check for unauthorized access attempts
libern --audit-failed-auth --since "24 hours ago"

# Review permission changes
libern --audit-permission-changes --since "7 days ago"

# Export monitoring report
libern --monitoring-report --output /reports/monthly/
```

### 7.2 Monitoring Dashboards

Organizations can build monitoring dashboards using:
- Libern's built-in status indicators
- Custom scripts parsing .aioss exports
- SIEM integration via log forwarding
- Grafana or similar visualization tools with database access

### 7.3 Alerting

Configure alerts for:
- Failed integrity checks (potential tampering)
- Multiple failed authentication attempts
- Unexpected peer authorization changes
- Extended peer disconnection
- Database integrity check failures

## 9. Auditor Communication

### 9.1 Providing Evidence to Auditors

When an auditor requests evidence for SOC 2:

1. **System description:** Provide the Libern architecture documentation
2. **Control descriptions:** Use the mapping in Section 2
3. **Operating evidence:** Provide .aioss ledger exports for the audit period
4. **Configuration:** Provide Libern configuration files
5. **Policies:** Provide related organizational policies

### 9.2 Auditor FAQ

**Q: Is Libern a SaaS platform?**
A: No. Libern is locally-deployed software. There is no multi-tenant cloud
infrastructure. The system boundary is the user's endpoint device.

**Q: How is access control enforced?**
A: Through Ed25519 cryptographic key pairs. Each user has a unique key pair.
Access to servers and channels is controlled by the server administrator.

**Q: How is audit logging implemented?**
A: The .aioss ledger records every operation with cryptographic proof. Each
entry is signed, timestamped, and hash-chained to prevent tampering.

**Q: How is encryption ensured?**
A: Data at rest is protected by platform-level encryption (BitLocker, FileVault,
LUKS). Data in transit is encrypted via P2P TLS channels.


## 11. SOC 2 Control Testing Procedures

### 11.1 Testing Access Controls (CC6.1)

```bash
# Test 1: Verify unauthorized peer cannot access data
libern --add-peer --peer-key INVALID_KEY
# Expected: "ERROR: Invalid Ed25519 public key"

# Test 2: Verify RBAC enforcement
libern --list-permissions --user non-admin-user
# Expected: Read-only access to limited channels

# Test 3: Verify cryptographic identity
libern --verify-identity
# Expected: "Identity: VERIFIED"
```

### 11.2 Testing Audit Logging (CC7.2)

```bash
# Test 1: Verify every operation generates an audit entry
libern --send-message "Test audit entry"
libern --aioss-list /path/to/ledger.aioss | tail -5
# Expected: Recent entry includes the test message

# Test 2: Verify hash chain integrity
libern --verify-ledger /path/to/ledger.aioss
# Expected: "Hash chain: VALID"

# Test 3: Verify tampering detection
libern --verify-ledger /path/to/tampered.aioss
# Expected: "Hash chain: BROKEN at entry 42"
```

### 11.3 Testing Availability (A1.1)

```bash
# Test 1: Offline mode
libern --send-message "Offline test"
# Expected: Message queued locally

# Test 2: Peer recovery
libern --peer-status
# Expected: Shows connected peers

# Test 3: CRDT convergence
libern --check-sync-status
# Expected: All peers synchronized
```
