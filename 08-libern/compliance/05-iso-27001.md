# ISO 27001 Mapping

## 1. Overview

ISO/IEC 27001 is the international standard for Information Security Management
Systems (ISMS). It specifies requirements for establishing, implementing,
maintaining, and continually improving an ISMS. Organizations that deploy Libern
as part of their infrastructure need to understand how the software supports
their ISO 27001 certification efforts.

This document maps Libern's features and architecture to the relevant ISO 27001
controls from Annex A, organized by the standard's four domains: Organizational,
People, Physical, and Technological controls.

## 2. ISO 27001:2022 Annex A Control Mapping

### 2.1 Organizational Controls (A.5)

#### A.5.1 — Policies for Information Security
Libern supports information security policy by providing a system with:
- Documented security features (encryption, authentication, audit logging)
- Predictable and verifiable behavior through open source
- Clear boundaries of responsibility (shared responsibility model)

#### A.5.2 — Information Security Roles and Responsibilities
- Libern's key-based identity model enables clear assignment of roles
- Server administrators have defined responsibilities
- Role-based access control supports separation of duties

#### A.5.8 — Information Security in Project Management
- Libern's development follows open-source security practices
- Code reviews, security testing, and vulnerability management processes
- Release signing and verification procedures

#### A.5.9 — Inventory of Information and Other Associated Assets
- Libern's data model is well-documented (SQLite schema, .aioss format)
- Asset classification can be implemented through server/channel naming
- Cryptographic identifiers enable asset tracking

#### A.5.10 — Acceptable Use of Information
- Server administrators define acceptable use policies
- Role-based access controls enforce usage restrictions
- .aioss audit trail records all usage for review

#### A.5.11 — Return of Assets
- When a user leaves, their Ed25519 key can be revoked
- Local data remains but can be deleted or exported
- Server administrator can remove the peer from all servers

#### A.5.12 — Classification of Information
- Channels and servers can be labeled based on information classification
- Access controls can be configured per classification level
- .aioss exports preserve classification metadata

#### A.5.15 — Access Control
- Cryptographic access control through Ed25519 key pairs
- Role-based permissions within servers
- Verifiable access control decisions through audit logs

#### A.5.16 — Identity Management
- Unique Ed25519 key pair for each user
- No shared identities or group accounts
- Key rotation procedures supported

#### A.5.17 — Authentication Information
- Private keys are the authentication information
- Stored locally in the user's key store
- Protected by OS-level security (keychain, credential manager)

#### A.5.18 — Access Rights
- Server owner grants and revokes access rights
- Role-based permissions manage granular access
- All access right changes are recorded in the .aioss ledger

#### A.5.19 — Information Security in Supplier Relationships
- Libern has minimal supplier dependencies (Rust toolchain, SQLite, OS)
- Open source supply chain is verifiable
- No external data processing suppliers

#### A.5.20 — Addressing Information Security Within Supplier Agreements
- When using Libern's enterprise support, agreements should include security terms
- Open source license (AGPL-3.0) provides baseline protections

#### A.5.21 — Managing Information Security in the ICT Supply Chain
- Libern binaries are signed and verifiable
- Source code is available for inspection
- Dependency vulnerabilities are tracked and patched

#### A.5.22 — Monitoring, Review, and Change Management of Supplier Services
- Libern updates are distributed via signed channels
- Changelogs document security-relevant changes
- Version pinning enables controlled rollouts

#### A.5.23 — Information Security for Use of Cloud Services
- Libern does not require cloud services
- When used with cloud storage for backup, appropriate controls should apply

#### A.5.24 — Information Security Incident Management Planning and Preparation
- .aioss audit trail supports incident detection
- Incident response procedures should be established by the organization
- Key compromise procedures are documented (see key recovery documentation)

#### A.5.25 — Assessment and Decision on Information Security Events
- .aioss ledger provides the data for event assessment
- Cryptographic verification enables accurate event classification

#### A.5.26 — Response to Information Security Incidents
- P2P isolation: compromised peers can be removed
- Key rotation invalidates compromised credentials
- Audit trail provides forensic evidence

#### A.5.27 — Learning from Information Security Incidents
- Incident post-mortems supported by .aioss audit records
- Open source code allows root cause analysis

#### A.5.28 — Collection of Evidence
- .aioss ledger provides cryptographically signed evidence
- Hash chain ensures evidence integrity
- Standard tools can be used for forensic analysis

#### A.5.29 — Information Security During Disruption
- Offline-first architecture ensures continued operation during disruptions
- P2P redundancy provides alternative communication paths
- CRDT convergence restores consistency after disruption

#### A.5.30 — ICT Readiness for Business Continuity
- Libern's architecture provides inherent business continuity
- No single point of failure
- Rapid redeployment on any hardware

### 2.2 People Controls (A.6)

#### A.6.1 — Screening
- Background screening of administrators is an organizational responsibility
- Ed25519 key identity enables accountability

#### A.6.2 — Terms and Conditions of Employment
- Employment terms should address acceptable use of Libern
- Confidentiality obligations apply to data accessed through Libern

#### A.6.3 — Information Security Awareness, Education and Training
- Libern documentation supports security training
- Transparent architecture simplifies security education

#### A.6.4 — Disciplinary Process
- .aioss audit trail supports disciplinary investigations
- Non-repudiation through Ed25519 signatures

#### A.6.5 — Responsibilities After Termination or Change of Employment
- Key revocation on termination
- Peer removal from all servers
- Local data deletion or transfer

#### A.6.6 — Confidentiality or Non-Disclosure Agreements
- Libern architecture inherently supports confidentiality
- Access controls enforce confidentiality requirements

#### A.6.7 — Remote Working
- Libern is designed for remote and offline work
- Local data ensures productivity without connectivity
- P2P sync works across VPN or local network

#### A.6.8 — Information Security Event Reporting
- Users can detect and report security events through audit analysis
- Interface provides clear feedback on security-relevant operations

### 2.3 Physical Controls (A.7)

#### A.7.1 — Physical Security Perimeters
- Libern runs on organization-managed hardware
- Physical security is the organization's responsibility

#### A.7.2 — Physical Entry Controls
- OS-level authentication controls access to Libern data
- Disk encryption protects data at rest

#### A.7.3 — Securing Offices, Rooms, and Facilities
- Standard organizational physical security applies

#### A.7.4 — Physical Security Monitoring
- OS-level monitoring and access logs apply

#### A.7.5 — Protecting Against Physical and Environmental Threats
- Standard organizational disaster recovery applies

#### A.7.6 — Working in Secure Areas
- Libern can be used in air-gapped environments
- No connectivity required for core functionality

#### A.7.7 — Clear Desk and Clear Screen Policy
- Libern supports OS-level screen lock
- Sensitive data can be hidden via channel permissions

#### A.7.8 — Equipment Siting and Protection
- Libern has no special equipment requirements
- Runs on existing hardware

#### A.7.9 — Security of Assets Off-Premises
- Laptop encryption protects Libern data off-premises
- P2P sync secures data in transit

#### A.7.10 — Storage Media
- .aioss export to removable media is supported
- Cryptographic integrity verification on export

#### A.7.11 — Supporting Utilities
- Libern requires only power and optional network connectivity
- No dependency on external utilities

#### A.7.12 — Cabling Security
- Standard network security practices apply
- P2P traffic is encrypted regardless of cable security

#### A.7.13 — Equipment Maintenance
- Standard organizational maintenance procedures apply

#### A.7.14 — Secure Disposal or Re-Use of Equipment
- Data can be securely deleted before equipment disposal
- .aioss export allows data preservation before wipe

### 2.4 Technological Controls (A.8)

#### A.8.1 — User Endpoint Devices
- Libern supports endpoint security best practices
- OS-level protections (encryption, authentication) are leveraged

#### A.8.2 — Privileged Access Rights
- Server owner role has elevated privileges
- Privileged operations are recorded in audit log
- Principle of least privilege is supported through RBAC

#### A.8.3 — Information Access Restriction
- Access controls at server and channel level
- Cryptographic enforcement of access restrictions
- Verifiable through .aioss ledger

#### A.8.4 — Access to Source Code
- Libern source is publicly available under AGPL-3.0
- Organizations can inspect, fork, and modify the code
- Development tools are standard (Rust, TypeScript)

#### A.8.5 — Secure Authentication
- Ed25519 cryptographic authentication
- No passwords stored or transmitted
- Key strength: 256-bit security level

#### A.8.6 — Capacity Management
- Libern storage scales with local disk capacity
- Processing scales with local CPU/GPU
- Performance characteristics are documented

#### A.8.7 — Protection Against Malware
- Libern is a single binary with minimal attack surface
- Code signing enables integrity verification
- Organizations should deploy endpoint protection per policy

#### A.8.8 — Management of Technical Vulnerabilities
- Open source enables community vulnerability discovery
- Release process includes security patches
- SBOM is available for dependency tracking

#### A.8.9 — Configuration Management
- Libern configuration is minimal and documented
- Configuration files can be managed through standard tools
- Default configuration is security-focused

#### A.8.10 — Information Deletion
- Local deletion is immediate
- .aioss tombstones mark deletion in the ledger
- Secure deletion depends on OS-level wipe

#### A.8.11 — Data Masking
- Channel permissions enable data masking
- Role-based access control hides sensitive content
- Not directly applicable to most Libern use cases

#### A.8.12 — Prevention of Data Leakage
- No automatic data transmission
- User-controlled peer authorization
- Encrypted P2P channels prevent interception
- No telemetry or analytics exfiltration

#### A.8.13 — Information Backup
- .aioss export serves as verifiable backup
- SQLite database can be backed up with standard tools
- P2P redundancy provides distributed backup

#### A.8.14 — Redundancy of Information Processing Facilities
- P2P architecture provides inherent redundancy
- Each peer maintains complete state
- Offline mode ensures continued processing

#### A.8.15 — Logging
- Comprehensive logging via .aioss ledger
- Cryptographic signature on every log entry
- Tamper-evident hash chain
- Exportable in machine-readable format

#### A.8.16 — Monitoring Activities
- .aioss ledger enables continuous monitoring
- Automated verification of log integrity
- Alerting can be built on log analysis

#### A.8.17 — Clock Synchronization
- Timestamps synchronized via NTP at the OS level
- Libern uses monotonic clocks for CRDT ordering

#### A.8.18 — Use of Privileged Utility Programs
- Libern has no privileged utilities
- All operations use standard system APIs

#### A.8.19 — Installation of Software on Operational Systems
- Libern is a single binary installation
- Signed releases enable verification
- Update process is controlled and auditable

#### A.8.20 — Networks Security
- Libern uses encrypted P2P channels
- mDNS discovery is LAN-only by default
- No exposure to public internet required

#### A.8.21 — Security of Network Services
- Libern does not depend on external network services
- P2P discovery uses standard network protocols

#### A.8.22 — Segregation of Networks
- Libern can be deployed on isolated network segments
- P2P traffic stays within the configured network

#### A.8.23 — Web Filtering
- Libern does not make HTTP requests to external services
- Web filtering configuration is irrelevant

#### A.8.24 — Use of Cryptography
- Ed25519 for digital signatures and identity
- SHA-3 for hash chaining
- TLS-equivalent for P2P transport encryption
- FIPS-compatible algorithms available

#### A.8.25 — Secure Development Life Cycle
- Libern follows secure coding practices
- Code reviews and testing before release
- Vulnerability management process

#### A.8.26 — Application Security Requirements
- Security requirements are documented in this compliance set
- Threat modeling is available in the research directory

#### A.8.27 — Secure System Architecture and Engineering Principles
- Defense in depth: encryption, signatures, access control, audit logging
- Least privilege: role-based access control
- Fail secure: offline-first, P2P redundancy

#### A.8.28 — Secure Coding
- Rust language provides memory safety
- TypeScript frontend follows secure coding guidelines
- Static analysis and testing in CI pipeline

#### A.8.29 — Security Testing in Development and Acceptance
- Unit tests, integration tests, security tests
- Fuzz testing for cryptographic components
- Regular dependency scanning

#### A.8.30 — Outsourced Development
- Libern is developed in-house by the core team
- Third-party dependencies are managed with SBOM
- Each dependency is evaluated for security

#### A.8.31 — Separation of Development, Test, and Production Environments
- Standard CI/CD separation for Libern development
- Organizations should maintain separate test environments

#### A.8.32 — Change Management
- Release process includes change review
- Changelogs document all changes
- Version pinning enables controlled deployment

#### A.8.33 — Test Information
- Test data is synthetic and not derived from production
- Open source test suite for verification

#### A.8.34 — Protection of Information Systems During Audit Testing
- .aioss ledger provides audit trail without impacting operations
- Read-only audit capability through ledger inspection

## 3. Statement of Applicability

Organizations implementing ISO 27001 with Libern should create a Statement of
Applicability (SoA) that includes:

- The controls from Annex A that are applicable to the Libern deployment
- Whether each control is implemented or not, with justification
- How the control is implemented through Libern's features and organizational
  policies

This document provides a starting point for that SoA. The organization's ISMS
documentation should reference this mapping and adapt it to specific deployment
configurations.

## 4. Risk Assessment Support

Libern supports the ISO 27001 risk assessment process by:

1. **Identifying assets:** The data model (SQLite, .aioss, keys) clearly
   defines information assets
2. **Identifying threats:** The open-source codebase enables thorough threat
   analysis
3. **Identifying vulnerabilities:** Transparent architecture simplifies
   vulnerability assessment
4. **Assessing risk:** The deterministic behavior of Libern enables accurate
   risk calculation
5. **Selecting controls:** This document maps controls to Libern capabilities
